## Finding 1 — digest cron runtime (FIXED)

The 00:30 UTC run's digest rows span 00:30:16 → 00:35:36 (~5.5 min observed, matrix
incomplete). Root causes measured from row timestamps:

- A single RU military digest took **~3m40s** (00:31:10 → 00:34:50) — TPM-throttled
  gpt-4o-mini retries on the 100-doc Cyrillic batch.
- The serial matrix is 2 dates × ~7 active countries × up to 3 tracks. Worst case
  comfortably exceeds even `maxDuration = 800`.
- The run proceeds alphabetically, so a killed run **silently drops the theaters that
  sort last — ua, a flagship validated theater, lost its today-digest** (1 qualifying doc
  existed at 00:35; no digest row, no error visible). Every 6h run had the same exposure.

**Fix shipped:** `?group=core|gulf` param on the digest route; vercel.json now runs
`group=core` (ru+ua) at :30 and `group=gulf` (all other active theaters) at :50 of hours
0/6/12/18. Each group's worst case fits the duration budget with headroom; a failure in
one group no longer starves the other.

Note: >300s execution is proven working in prod (rows landed at ~330s elapsed), so the
account honors `maxDuration = 800` — the split is for headroom and blast-radius isolation,
not because of a 300s plan cap.
