## TASK 1 — Adapter

Behind the existing `SourceAdapter` interface, replacing the `telegram_mtproto` stub:

1. **Per-invocation lifecycle:** connect → fetch → disconnect inside one cron invocation, strict internal time budget under the route's `maxDuration`. No long-lived connections in serverless.
2. **Peer caching:** `ResolveUsername` is one of Telegram's most tightly flood-limited calls. Resolve each channel once, persist peer id + access_hash (registry meta or a small table), and use saved peers on every subsequent run. A resolve failure marks the channel for retry with backoff, not a hot loop.
3. **Incremental by high-water mark:** per-channel last-seen message id; fetch only newer; cap N messages/channel/run and M channels/run with rotation so backlog can never blow the duration. Persist marks so a crash resumes cleanly.
4. **Normalization:** RawDoc with text (or media caption), canonical URL `t.me/<channel>/<id>`, posted_at from the message date, the channel's existing sourceKey, original language preserved (no translation at ingest). Skip empty/media-only messages cheaply.
5. **No double-ingest:** a message already ingested via preview scrape must not create a second doc. Check how preview docs are keyed; ensure content-hash dedupe plus the `doc_dedup` gate catches cross-adapter duplicates, and add a test using a channel present in both paths.
6. **FLOOD_WAIT handling:** honor small waits in-run, abort-and-resume on large ones; count both in `cron_runs` counts jsonb.
7. **Theater tagging:** through `TELEGRAM_CHANNEL_THEATER` only — the standing ruling stands: never route by language; the Lebanese trio stays → ir.
8. Unit tests: high-water logic, peer-cache reuse, normalization, dedup exclusion, flood-wait abort path, cap enforcement.
